### 🎯 第一颗珍珠：执行器（Executor）—— 用户命令的翻译官
```text
SeqScanExecutor / FilterExecutor
```
- 它干什么？
    - 当你写 SELECT * FROM users WHERE age > 18，执行器就是那个真正去“一行一行捞数据、一行一行过滤”的人。它遵循火山模型：上层算子（比如 Limit）不停地喊 next()，下层算子（比如 Filter）就向再下层（SeqScan）要数据，像流水线一样把元组（Tuple）往上传递。

- 它依赖谁？
  - 它不碰磁盘，也不管索引。SeqScanExecutor 会直接问 TablePage 要数据；而如果有索引，IndexScanExecutor（虽然你还没写）会问 BPlusTree 要数据。

### 🎯 第二颗珍珠：索引（BPlusTree）—— 数据的快速导航仪
```text
BPlusTree
├── BPlusTreeInternalPage（内部节点，存路牌）
└── BPlusTreeLeafPage（叶子节点，存真实键值对）
```
- 它干什么？
  - 当你执行 WHERE id = 100 时，BPlusTree 就是那个从根节点出发，一路根据“路牌”向下找，最终精准定位到目标数据物理位置的导航员。

- 怎么找？
  - findLeafPage(key) 从根开始，不断调用 BufferPoolManager.fetchPage(pageId) 获取内部节点页面，再用 lookup(key) 判断下一步走哪个儿子，直到抵达叶子节点。

- 插入数据时呢？
  - insert(key, value) 会先找到叶子节点，把键值对塞进去。如果叶子满了，就调用 splitLeafPage 劈成两半，并把分裂产生的新键“提拔”到父节点（insertIntoParent）。

- 它依赖谁？
  - 它完全不碰磁盘，所有页面都是向 BufferPoolManager 要的。

### 🎯 第三颗珍珠：堆表页（TablePage）—— 数据的真正容器
```text
TablePage（分槽页）
```
- 它干什么？
    - 如果说 BPlusTree 存的是“索引 → 物理地址”的映射，那 TablePage 存的就是真实的行数据。它在一个 4KB 页面里用“槽位数组 + 数据区”的方式，高效地管理多条变长记录（Tuple）。

- 怎么插入一行？
  - insertTuple(tuple) 会检查空闲空间够不够，然后把元组的字节数组拷贝到页面底部，并在顶部槽位数组里记下“偏移量 + 长度”。

- 怎么读一行？
  - getTuple(slotId) 直接根据槽位记录的偏移量，从字节数组里精准抠出数据，组装成 Tuple 返回。

- 逻辑删除
  - deleteTuple(slotId) 仅仅把槽位里的偏移和长度清零，数据还在原地，只是“看不见”了。

- 它依赖谁？
  - 它也不碰磁盘。它只是工具人，专门操作 Page 对象里的 byte[] data。

### 🎯 第四颗珍珠：缓冲池（BufferPoolManager）—— 内存大管家
```text
BufferPoolManager
├── pages（Page[] 数组，真实内存）
├── pageTable（Map<Integer, Integer>，pageId → 数组下标）
└── replacer（LRUCache，决定淘汰谁）
```
- 它干什么？
它是整个系统的交通枢纽。上层不管要什么页面（索引页、表数据页），都必须通过它来获取。它手里握着一个固定大小的 Page[] 数组，这就是内存缓冲池的本体。

- fetchPage(pageId) 的逻辑： 
  - 1.先查 pageTable：如果在内存，直接 pin 住返回（命中）。 
  - 2.如果不在，找空槽位；没有空槽位就喊 replacer.victimize() 踢走一个最久没用的。 
  - 3.被踢走的如果是脏页，先让 DiskManager 写回磁盘。 
  - 4.让 DiskManager 把目标页从磁盘读入该槽位。 
  - 5.更新页表，pin 住，返回。
- unpinPage(pageId, isDirty)：
  用完后减少引用计数。如果计数归零，就把页面丢进 replacer 的候选队列，允许被淘汰。如果数据被改过，打上脏标记。

- 它依赖谁？
  - DiskManager：干所有磁盘 I/O 的苦力活。
  - LRUCache：帮它决定淘汰谁。

### 🎯 第五颗珍珠：磁盘管理器（DiskManager）—— 物理硬盘的苦力
```text
DiskManager（操作 cyber.db 文件）
```
- 它干什么？
    - 真正和操作系统文件打交道的家伙。它用 RandomAccessFile 打开 cyber.db，根据 pageId 计算出文件偏移量（pageId × 4096），执行读写。

- readPage(pageId, page)：
  - 把文件指针挪到对应位置，读 4KB 填进 page.getData()。如果读的位置超过文件末尾（新页），就返回全零的干净页。

- writePage(pageId, page)：
  - 把 page.getData() 的 4KB 原样写回文件对应位置，并清除脏标记。

- allocatePage()：
  - 返回一个递增的 pageId，不实际分配磁盘空间（延迟到第一次写入时由操作系统分配）。

- 它依赖谁？
  - 谁也不依赖，它是最底层。

### 🎯 第六颗珍珠：事务相关（LockManager & LogManager）—— 安全与恢复双保险
- LockManager（保安队长）
  - 职责：管理行级读写锁（S锁 / X锁），保证多个事务并发时数据不乱套。
  - 用法：读数据前 acquireSharedLock(recordId)，写数据前 acquireExclusiveLock(recordId)，用完释放。

- LogManager（日记本）
  - 职责：实现 WAL（预写式日志）。任何修改操作，必须先写日志，再改内存页。
  - appendLogRecord(record)：把操作记录追加到日志（模拟落盘）。
  - recover()：崩溃重启后，重放日志恢复数据。

### 🎯 第七颗珍珠：元组（Tuple）与页面（Page）—— 数据的原子单位
- Tuple（一行数据）
    - 就是一个包装了 byte[] 的不可变对象，代表一行数据。执行器、TablePage、索引都围绕它转。

- Page（4KB 内存页）
  - 缓冲池数组里存的就是它。它有 pageId、data 字节数组、isDirty 脏标记、pinCount 引用计数。
  - 它是 DiskManager 和 BufferPoolManager 之间传递数据的唯一货币。

#### 🔗 完整串联：一次 SELECT 查询的旅程
```text
用户调用 BPlusTree.getValue(100)
│
▼
BPlusTree.findLeafPage(100)  
│  循环：要内部节点页面
▼
BufferPoolManager.fetchPage(pageId)
│  1. 查 pageTable
│  2. 若无，找空槽或 LRU 淘汰
│  3. DiskManager.readPage()
▼
DiskManager 从 cyber.db 读取 4KB 到 Page.data
│
▼
BPlusTree 拿到叶子页，遍历键值对找到 value（物理地址）
│
▼
（如果是查真实行数据）TablePage.getTuple(slotId)
│  从 Page.data 中拷贝出 Tuple 字节
▼
返回 Tuple 给上层执行器
```
#### 🔗 完整串联：一次 INSERT 的旅程
```text
用户调用 BPlusTree.insert(100, value)
│
▼
LockManager.acquireExclusiveLock(100)   ← 先加写锁
│
▼
LogManager.appendLogRecord(record)      ← 先写日志
│
▼
BPlusTree.findLeafPage(100) → 拿到叶子页
│
▼
leafPage.insert(key, value)             ← 修改索引页（内存）
│  如果满了 → splitLeafPage → insertIntoParent
│  每改一页都 page.setDirty(true)
▼
（同时，行数据插入 TablePage，类似过程）
│
▼
BufferPoolManager.unpinPage(..., true)  ← 释放页面，标记脏页
│
▼
LockManager.releaseExclusiveLock(100)   ← 释放写锁
│
▼
（未来某个时刻）BufferPoolManager 淘汰脏页
│
▼
DiskManager.writePage(pageId, page)     ← 真正写回磁盘
```
### 💡 一句话记住所有类的关系
- 执行器向索引要位置，索引向缓冲池要页面，缓冲池问LRU淘汰谁、然后让磁盘管理器读写文件；锁管理器管并发，日志管理器保安全；TablePage 和 Tuple 是数据的容器，Page 是一切 I/O 的单位。
